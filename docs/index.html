1. Introduction

(a) End goal

Tidyer is a tabletop manipulation system that perceives cluttered objects with a depth camera, reasons about grasps and arm motion, and uses a UR-series robot arm (via MoveIt) to pick and place items. The end goal is reliable, repeatable tidy-up behavior suitable for a lab demo and future extension to more complex scenes.

(b) Why this project is interesting

Tabletop tidying combines perception, planning, and control under real sensor noise, calibration error, and limited compute. Interesting problems include: segmenting and localizing objects in dense RGB-D data, producing stable pick poses, coordinating gripper timing with motion, and keeping the software stack modular enough to iterate quickly during integration weeks.

(c) Real-world applications

Similar pipelines appear in warehouse piece picking, kitting cells, assistive robots that clear surfaces, recycling sort lines, and light assembly where cameras replace fixturing. The same building blocks—3D perception, motion planning, and guarded execution—transfer directly to those domains.

2. Design

(a) Design criteria and desired functionality

  • Perceive the workspace from a fixed camera mount and publish actionable geometry to the planner.
  • Plan collision-aware arm trajectories using the standard UR + MoveIt stack.
  • Execute pick–place sequences with a controllable gripper and safe shutdown if a node exits.
  • Remain debuggable: separate nodes for perception, transforms, and manipulation; keyboard trigger in its own process.

(b) Architecture we chose

A ROS 2 ament_python package (tidyer) hosts perception and planning nodes. Bring-up is centralized in a single launch file that starts RealSense (point cloud + aligned depth), perception, static transforms, the pick–place executor, and MoveIt. Interactive input runs in a separate terminal so it retains a TTY.

(c) Design choices and trade-offs

  • RGB-D + point cloud vs. contact or fiducials: richer scene understanding at the cost of tuning and lighting sensitivity.
  • MoveIt / UR defaults vs. custom controllers: faster integration vs. less low-level control.
  • Modular nodes vs. one monolithic process: clearer failure modes but more timing and message contracts to maintain.

(d) Impact on robustness, durability, and efficiency

Perception-heavy designs trade robustness to occlusion and reflectivity for hardware simplicity (no part-specific nests). MoveIt-based motion improves safety and repeatability in structured cells. Multi-node ROS graphs add latency and integration overhead but improve testability and team parallelization—important for a short course timeline.

  • Gripper with a simple binary open/close service (as wired in software).
  • Table, fixturing, and any 3D-printed or machined brackets—document with CAD links in section 7.

(c) Software

Key modules in this repository:

  • tidyer/perception/process_pointcloud.py — depth / point cloud processing pipeline.
  • tidyer/planning/main.py — pick–place node: IK planning, trajectory execution, gripper services, job queue.
  • tidyer/planning/ik.py — inverse kinematics helper used by the planner.
  • tidyer/planning/static_tf_transform.py — static transforms for consistent frames.
  • tidyer/planning/keyboard_trigger.py — interactive triggers (run separately from launch).
  • launch/tidyer_bringup.launch.py — starts RealSense, perception, TF, pick–place, and MoveIt; global shutdown if any launched process exits.

(d) End-to-end system flow

Launch (tidyer_bringup.launch.py)
  ├─ RealSense (point cloud + aligned depth)
  ├─ process_pointcloud  →  perception outputs (topics TBD in your write-up)
  ├─ tidyer_tf           →  static / supporting transforms
  ├─ tidyer_pick_place   →  subscribes to pick pairs, plans/executes trajectories, gripper
  └─ MoveIt (UR)         →  planning scene, RViz optional

Separate terminal: ros2 run tidyer tidyer_keyboard  (interactive TTY)

(b) Difficulties

There were several challenges we encountered in both the hardware and software.

  • TF Tree Issue: We encountered several challenges throughout our project, including a disconnected TF tree between the base_link and camera frames that blocked necessary pose transformations.
  • Planner Failures: We experienced frequent planner failures and timeouts, particularly when the target was far away or the environment was heavily cluttered.
  • Perception Sensitivity: Our perception system showed high sensitivity, as HSV thresholds drifted under changing lighting conditions, which affected segmentation accuracy.
  • Grasp/Place Issues: Excessive noise in centroid and depth data negatively impacted our grasp and place accuracy.
  • Robot Protection Stops: The UR robot frequently entered a protective stop, caused by compounding errors in depth/pose estimation combined with overly aggressive grasping and approach tuning in certain code branches.
  • Table Alignment: The table was not perfectly straight, introducing consistent calibration errors.
  • Camera Discrepancy: We noted a significant discrepancy between the optimal camera placement and the actual depth data received from the PyRealSense camera.
